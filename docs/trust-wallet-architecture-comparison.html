No blockchain required

Supabase is enough. The existing schema is not.

For an app that looks and behaves like Trust Wallet using mock currency, Supabase/Postgres is a good backend. You do not need chains, keys, gas RPCs or cryptographic signing. However, the present database only supports internal transfers between two wallets owned by the same user and immediately marks them completed.

Recommendation: keep Supabase, upgrade the ledger.Build a secure internal double-entry ledger with mock wallet addresses, cross-user recipient lookup, pending/completed/failed states, idempotency, reserved funds, immutable entries and Realtime history updates.

1. What the mock app should resemble

“Realistic” here means reproducing the user experience and transaction semantics, while the balance remains entirely inside your database.

User-visible behavior

  • Every user has one or more named wallets.
  • Each wallet has a unique shareable mock address and QR code.
  • A sender can paste/scan an address or choose a handle.
  • The review screen shows amount, recipient and a clearly labeled mock fee.
  • History shows Sent/Received, pending, completed or failed.
  • The receiver sees the incoming transfer through Realtime.

Backend behavior

  • Only the server-side transaction operation changes balances.
  • Sender funds are checked and reserved atomically.
  • Every transfer creates balanced debit and credit ledger entries.
  • Retries cannot charge a user twice.
  • Both parties can read the same transaction but no unrelated user can.
  • An admin faucet controls creation of mock currency.
Important: call the assets “demo” or “mock” throughout the app. Do not imply that the values are redeemable cryptocurrency or that the mock transaction ID exists on a public network.

Why the existing transfer cannot work between users

The function transfer_between_mock_wallets checks that both p_from_wallet_id and p_to_wallet_id belong to auth.uid(). Alice therefore cannot send to Bob. It is currently a transfer between Alice’s own “Main Wallet” and Alice’s own “Savings Wallet.”

3. What must improve

PriorityGapRequired change
P0Cross-user recipientsAdd a unique public mock address per wallet. The transfer operation accepts a recipient address, resolves the destination privately, and verifies only the source belongs to the caller.
P0No immutable ledgerAdd double-entry ledger_entries. A transfer produces a sender debit and receiver credit. Never edit or delete posted entries; correct mistakes with reversal entries.
P0Unsafe write fallbackRemove local fallback for funding and transfers. A database failure must show “not sent.” Local fixtures may remain only in an explicit demo/offline build.
P0Anyone can mint demo moneyThe current authenticated funding RPC lets each signed-in user add arbitrary funds. Replace it with an admin-only faucet, fixed starter grant or rate-limited daily claim.
P0No retry protectionRequire a client-generated UUID idempotency_key, unique per sender. Returning the same request must return the original transaction—not debit twice.
P1No lifecycleAdd created, pending, completed, failed, cancelled and reversed. Reserve sender funds while pending.
P1No receiver visibilityStore both sender and receiver owner IDs on the protected transaction row and permit SELECT when either equals auth.uid().
P1No address/handle privacy designExpose only mock address, unique handle, display name and avatar in recipient search. Never expose emails, balances or internal user UUIDs.
P1Weak amount representationUse integer base units in Postgres and strings/BigInt in TypeScript. Avoid JavaScript Number for ledger arithmetic.
P2No realistic transaction metadataAdd a generated display reference, note/memo, fee, timestamps, failure reason and balance snapshots. Label references as mock.
P2No operational controlsAdd transfer limits, rate limits, blocked accounts, audit events, reconciliation queries and admin reversal workflow.

Why keep balances and ledger entries?

wallet_balances makes the home screen fast. ledger_entries makes the system auditable. A reconciliation query can recompute every balance from immutable entries and detect corruption. Only the transfer/faucet/reversal operations may change the balance projection.

Mock transaction reference: generate something visually familiar, such as mtx_01J... or a 64-character hex string, but label it “Mock transaction ID.” Do not call it an on-chain hash.

5. Realistic user-to-user mock transfer

1 · RecipientBob shares his mock address/QR or Alice searches his handle.
→
2 · ReviewAlice selects asset/amount and sees Bob’s masked address and mock fee.
→
3 · InitiateServer verifies Alice, resolves Bob, checks limits and reserves funds atomically.
→
4 · PendingBoth users receive the same pending transaction through Realtime.
→
5 · SettleWorker posts debit/credit/fee entries and changes status to completed.
→
6 · HistoryAlice sees Sent; Bob sees Received; balances reconcile to the ledger.

Atomic initiation rules

  1. Authenticate a permanent user and validate source-wallet ownership.
  2. Return the original result if the idempotency key already exists.
  3. Resolve destination by exact mock address; do not expose its owner UUID.
  4. Lock the relevant sender balance row.
  5. Check available ≥ amount + fee, wallet status and transfer limits.
  6. Move amount + fee from available to reserved and insert one pending transaction.
  7. Commit everything together or commit nothing.

Settlement rules

OutcomeAccounting actionUser experience
CompletedConsume sender reserve; write debit and receiver credit; route fee to a system wallet or record a balanced fee entry.Both histories update; receiver balance increases.
FailedRelease the entire reserved amount to sender; no receiver credit.Both see failed if the pending transaction was already shared; reason is user-safe.
CancelledRelease reserve before settlement.Sender sees cancelled; receiver does not receive funds.
ReversedCreate a new linked reversal transaction and opposite ledger entries—never edit the original.History preserves both original and reversal.
Simpler MVP option: settle immediately in one RPC and animate “Processing” for 1–2 seconds only in the UI. This avoids a worker and reserved balances. If you show a true persisted Pending state, however, implement real reservation and recovery logic; never use a timer alone.

6. Supabase security changes

  • Use permanent email/OAuth accounts for user-to-user value. Anonymous Supabase users receive the authenticated database role but can lose the identity when local credentials disappear.
  • Allow clients to SELECT only their own wallets/balances and transactions where they are sender or receiver.
  • Deny direct client INSERT/UPDATE/DELETE on balances, transactions and ledger entries.
  • Perform transfers through a trusted Edge Function or narrowly exposed RPC; never ship the service-role key in the Expo app.
  • Prefer internal tables/functions in a non-exposed schema. Supabase specifically warns against exposed SECURITY DEFINER functions.
  • Explicitly revoke default function execute privileges and grant only the intended API role.
  • Rate-limit recipient lookup and transfer initiation to prevent handle enumeration and spam.
  • Test RLS as Alice, Bob, an unrelated user, an anonymous Auth user and an unauthenticated request.
  • Store authorization flags in protected app metadata/database roles, never user-editable metadata.

Final assessmentThe current database is approximately the right technology and a useful prototype, but it is not enough for user-to-user mock transfers. The smallest credible upgrade is: public mock addresses + cross-user atomic RPC + idempotency + immutable ledger + restricted faucet + no local write fallback. Pending status, reserved funds, fees and Realtime then provide the Trust Wallet-like experience.
