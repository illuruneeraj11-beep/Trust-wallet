— call walkthrough

The app is finished, tested, and pushed to GitHub. The only thing left needs the Expo account that owns the project (your friend's login). This page is the script for that call.

Done  All code, tests, assets, database migrations, and live APIs work. Everything is pushed to github.com/illuruneeraj11-beep/Trust-wallet (branch main).

Blocked  Expo's build service (EAS) will only build for the account that owns project 463c7e8a-2dbd-4ba2-9203-46d4e297bc10. Steps to do on the call (his computer)

Pull the latest code

In a terminal, inside his project folder:

git pull origin main

If he doesn't have the folder: git clone https://github.com/illuruneeraj11-beep/Trust-wallet.git

Enter the app folder and install packages

cd expo-wallet
npm ci

Takes a few minutes. Needs Node.js installed (it was, if he built before).

Log in to HIS Expo account

npx eas-cli logout
npx eas-cli login

He types his Expo username and password (same account as his earlier APK attempt). Then verify:

npx eas-cli whoami
npx eas-cli project:info

Checkpoint: project:info must print project details with no error. If it says "Entity not authorized", it's the wrong account — log in with the other one. If he forgot the password: reset at expo.dev.

Add the 4 build settings (exact values — paste one line at a time)

These tell the installed APK where the backend and live prices live. All are safe public values.

npx eas-cli env:create --environment preview --name EXPO_PUBLIC_WALLET_MODE --value connected --visibility plaintext
npx eas-cli env:create --environment preview --name EXPO_PUBLIC_SUPABASE_URL --value https://bgwsoyfsyoecsuemckel.supabase.co --visibility plaintext
npx eas-cli env:create --environment preview --name EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY --value sb_publishable_HX8YGOPXCZEh-hlTfrC10Q_IoPahOxz --visibility sensitive
npx eas-cli env:create --environment preview --name EXPO_PUBLIC_MARKET_API_BASE_URL --value https://expo-wallet.vercel.app --visibility plaintext

Verify all four exist:

npx eas-cli env:list --environment preview

"Already exists" errors are fine — skip that one.

Start the build

npx eas-cli build --platform android --profile preview
  • If asked "Generate a new Android Keystore?" → Yes.
  • The build runs in Expo's cloud, ~10–20 minutes. He can close nothing — just wait.
  • At the end it prints a link like https://expo.dev/accounts/…/builds/… — that link IS the APK download. Have him send it to you.

Install and test

Open the build link on an Android phone → Install (allow "unknown sources" if asked). Test on two phones: create two accounts, fund wallets, send both directions, scan a QR, try airplane mode. Only share the link after that works.

How this actually works (for the curious)

How the APK gets made

Think of EAS Build as Vercel for mobile apps. When he runs eas build, the code is uploaded to Expo's cloud, where a build machine spins up — same idea as when you git push and Vercel/Render spins up a build container. The difference is what comes out the other end:

  • Vercel/Render: the build produces a website/server that stays on their machines, and users visit a URL. Deploy again → everyone instantly sees the new version.
  • EAS: the build produces a file — the .apk. It compiles your React Native TypeScript into JavaScript bundles, packages them with real native Android code (camera, secure storage, UI engine), and wraps it all into one installable package. That file is downloaded onto each phone. Deploy again → you get a new file, and every phone must install it again. Nothing "stays running" in the cloud.

The other new concept is signing. Android refuses to install unsigned apps, so EAS generates and stores a keystore — a digital signature proving every future build comes from the same publisher. That's why the account matters so much: the keystore lives inside his Expo account, and it has no equivalent in web deploys. (Closest analogy: the HTTPS certificate Vercel manages for you — except here losing it means phones treat the next build as a different app.)

And the env vars? Same idea as Vercel environment variables, with one twist: on Vercel they live on the server and can hold secrets. Anything named EXPO_PUBLIC_* gets baked into the APK file itself at build time — anyone can unzip an APK and read them. That's why only public values go there, and why changing one requires a rebuild, not just a redeploy.

How the app talks to Supabase

Exactly like your web apps — there is no special "mobile backend". The APK contains the Supabase JS client, and the phone makes HTTPS calls straight to https://bgwsoyfsyoecsuemckel.supabase.co:

  • Auth: email sign-in via Supabase Auth; the session token is kept in Android's encrypted SecureStore (the mobile version of an httpOnly cookie).
  • Ledger: wallets, balances, and transfers are rows in Postgres. The app never edits balances directly — it calls database functions (RPCs) that validate address, network, amount, and fees, then debit/credit atomically. Row-level security means a user can only ever see their own data, even though the API key is public.
  • Realtime: the recipient's phone holds a websocket subscription to Supabase, so when phone A sends, phone B's balance updates without refreshing — same feature you'd use in a web app.

So the "money" is just Postgres rows, which is why this is safely a mock: no blockchain, no keys, nothing real to lose.

Where your Vercel deployment fits — it's NOT just for web testing

The Vercel project (expo-wallet.vercel.app) does two jobs:

  • Web preview of the app — the same Expo code exported as a website. Handy for quick testing, yes.
  • Live price API for everyone, including the APK — the /api/markets and /api/perps routes run as serverless functions on Vercel (your familiar territory). The phone app calls them via EXPO_PUBLIC_MARKET_API_BASE_URL. This exists because the CoinMarketCap API key must stay secret, and as explained above an APK can't hold secrets — so the phone asks your Vercel server, and the server (which can hold the key) asks CoinMarketCap. A classic proxy, the same pattern you'd use in any web app to hide an API key.

So if the Vercel project is ever deleted or paused, installed APKs lose live prices (wallet features would still work — those hit Supabase directly).

If something goes wrong

  • "Entity not authorized" → wrong Expo account. npx eas-cli logout, log in with the account that created the project.
  • He can't recover the old Expo account at all → fallback: any account can adopt the app with npx eas-cli init --force (creates a fresh project under that account), then continue from step 4. Nothing is lost — this is a fresh mock app.
  • Build fails in the cloud → open the build link, copy the red error text, send it to me.
  • App installs but shows no data → step 4 was skipped or a value was typo'd; run env:list and compare against the table above, then rebuild.
Reminders: this is a testnet/mock app — no real money, keys, or crypto ever. Share the build link privately with testers only, and keep the "Testnet" name visible so nobody mistakes it for the official Trust Wallet.
